  Definition of Done (DoD) по этапам — для MVP “Гюмри гид”

  Этап 1. UX/тексты: “гео обязательно” (до маршрута)
  Готово, если:
   - Есть 3 подготовленных сообщения (RU и EN):
       1. Зачем нужна гео (например: “Чтобы построить маршрут от вашего текущего
          местоположения, мне нужна геолокация”).
       2. Как отправить гео (например: “Пожалуйста, отправьте её через меню 📎 →
          «Геопозиция»”).
       3. Что отвечать, если пользователь пишет текст вместо гео (например: “Я
          вас понял, но для построения маршрута мне всё же нужна геолокация.
          Пожалуйста, отправьте её.”).
   - Процесс выглядит так: пользователь описывает предпочтения, и сразу после
     этого бот просит прислать геолокацию, используя эти тексты.

  Этап 2. База мест (places_gyumri.json)
  Готово, если:
   - Проверено не менее 30–40 записей в places_gyumri.json вручную.
   - У этих записей есть корректные lat, lon, category (food, sight, museum),
     name_ru, name_en.
   - Удалены или исправлены записи без координат или названий.
   - При вызове get_nearby_places() для центра Гюмри возвращается как минимум
     5–10 релевантных мест в радиусе 1–2 км.
   - В базе есть хотя бы 10–15 мест с category: "food".

  Этап 3. Парсинг времени из текста
  Готово, если:
   - Создана функция (например, parse_time(text: str) -> Optional[float]),
     которая возвращает количество часов.
   - Функция корректно обрабатывает как минимум: "2 часа", "4h", "1 день" (→ 8
     часов), "полдня" (→ 4 часа).
   - UserState.preferences.time_available_hours заполняется результатом этой
     функции. Если время не найдено, значение остаётся None.

  Этап 4. FSM-переходы + “бутылочное горлышко” на гео
  Готово, если:
   - UserState.stage обновляется согласно FSM: NEW_USER → ASK_STYLE →
     ASK_PREFERENCES → ASK_LOCATION_REQUIRED.
   - На стадии ASK_LOCATION_REQUIRED:
       - Если приходит location, stage меняется на SHOW_PROGRAM_OPTIONS.
       - Если приходит text, бот отвечает сообщением "пришлите гео" и stage не
         меняется.
   - В коде нет пути, по которому stage может стать SHOW_PROGRAM_OPTIONS без
     получения геолокации.

  Этап 5. Генератор 2–3 программ от гео
  Готово, если:
   - Создана функция generate_programs(lat, lon, preferences) -> List[Program].
   - Функция возвращает 2–3 программы (например, в виде списка place_id).
   - Логика генерации: фильтрует места по category, сортирует по расстоянию от
     lat, lon, ограничивает число точек по time_available_hours.
   - На стадии SHOW_PROGRAM_OPTIONS бот показывает пользователю краткое описание
     этих 2–3 программ (с кнопками или номерами для выбора).

  Этап 6. ON_ROUTE (ведение по маршруту)
  Готово, если:
   - После выбора программы UserState.current_route и UserState.visited_places
     заполняются. stage меняется на ON_ROUTE.
   - Есть обработчики для команд/кнопок:
       - "Следующая": показывает инфо о первой точке из current_route, которой
         нет в visited_places.
       - "Пропустить": добавляет текущую точку в visited_places и показывает
         следующую.
   - Когда все точки посещены, бот сообщает о завершении тура.

  Этап 7. LLM-политика (контроль над "галлюцинациями")
  Готово, если:
   - До получения гео вызовы к LLM либо не происходят для конкретных советов,
     либо system_prompt жёстко запрещает упоминать названия/адреса.
   - При генерации программ и ведении по маршруту в модель передаётся конкретный
     список мест (кандидаты для программ или точки маршрута) с инструкцией:
     "Отвечай, опираясь только на этот список. Не придумывай другие места".
   - Ручная проверка 5–10 ответов модели показывает, что она не выдумывает
     несуществующие заведения.

  Этап 8. Ручные тесты
  Готово, если:
   - Составлен чек-лист из 5–8 ключевых сценариев.
   - Все сценарии пройдены вручную от начала до конца:
       1. RU, happy path: онбординг → предпочтения → гео → выбор программы →
          прошёл 2–3 точки → завершил.
       2. EN, happy path.
       3. "Непослушный" пользователь: на просьбу прислать гео пишет "нет" или
          задаёт вопрос → бот корректно повторяет просьбу.
       4. Пользователь на маршруте присылает новую гео → команда "что рядом"
          показывает актуальные места.
   - Все пройденные тесты завершились успешно и без "зависаний" логики.
